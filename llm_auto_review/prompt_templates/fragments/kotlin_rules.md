# Kotlin правила и частые ошибки

## Null-безопасность
- Не проверять NPE — Kotlin гарантирует null-безопасность через систему типов.
- `?.` и `!!` — используй `?.` для безопасного вызова, `!!` только если ты уверен, что null невозможен (и это подтверждено контрактом).

## Условия
- `==` vs `!=` — проверь, не перепутаны ли.
- `isEmpty()` vs `isNotEmpty()` — проверь направление.
- `isBlank()` vs `isNotBlank()` — проверь направление.
- `in` vs `!in` — проверь направление.
- `true`-ветка и `else`-ветка — не перепутаны ли.
- Ранний `return` — не ломает ли остальные сценарии.

## Подозрительные паттерны
```
if (hasAccess) { showDialog() }           // возможно инвертировано
if (isEnabled) { blockAccess() }          // возможно !isEnabled
if (isPremium) { showPaywall() }          // возможно !isPremium
return true при ошибке                    // возможно должен быть false
if (condition) return false               // проверь, не нужен ли return true
```

## Feature Flags
- Новый функционал за флагом — проверь, что не перепутан флаг.
- Флаг не должен закрывать старый код случайно.
- Проверь консистентность: если флаг добавлен в одно место, он должен проверяться во всех связанных.

## Строковые сравнения
- `.lowercase()` / `.uppercase()` — проверь, что обе стороны сравниваются в одном регистре.
- `.trim()` — проверь, что пользовательский ввод обрезается перед сравнением.
- `equals(ignoreCase = true)` — используй для case-insensitive сравнения.

## Access boolean: DEF → все USE
- После присваивания `hasAccess` / `isAvailable*` / `accessCondition` / `isPremium` — перечисли **все** `if`/`when` с этим именем в том же method.
- `[PASS]` на семантику присваивания **не** закрывает ожидание о доступе, пока не проверены все USE (см. `access_def_use.md`).
- Таблица комбинаций **других** флагов не заменяет проверку полярности одной переменной на разных строках.

## Поток управления
- Для блоков с 2+ флагами строй таблицу истинности (все комбинации).
- Трассировка пользователя: от точки входа до конечного экрана, шаг за шагом.
