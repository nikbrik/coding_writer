# evolve — Harness Evolution Skill

## Когда использовать
- После завершения большой фичи, рефакторинга, сложной отладки
- Когда агент был поправлен пользователем 2+ раза за сессию
- Когда найден нетривиальный workaround или неочевидное решение
- По запросу: `/evolve`, "evolve harness", "обнови harness", "что мы узнали"

---

## ФАЗА 1 — Сбор сигналов

Просмотри всю историю текущего разговора. Ищи сигналы пяти типов:

**Тип A — Коррекции пользователя** [приоритет HIGH]
- "нет", "не так", "стоп", "не используй X", "всегда делай Y"
- Пользователь отменил действие агента и объяснил почему
- Одна и та же правка применена 2+ раз подряд

**Тип B — Нетривиальные решения** [приоритет HIGH]
- Решение потребовало 3+ итераций или смены подхода
- Найден workaround для проблемы инструмента/библиотеки/окружения
- Обнаружена неочевидная зависимость между компонентами

**Тип C — Паттерны проекта** [приоритет MEDIUM]
- Повторяющаяся последовательность действий (3+ раз в сессии)
- Соглашение проекта, которого нет в AGENTS.md
- Архитектурное решение с объяснённым "почему"

**Тип D — Провалы с причинами** [приоритет MEDIUM]
- Команда упала — зафиксировать команду, симптом, причину, fix
- Тест провалился неожиданным образом
- Инструмент повёл себя иначе чем ожидалось

**Тип E — Новые находки** [приоритет LOW]
- Более эффективный способ делать что-то стандартное
- Полезная команда/утилита для этого проекта
- Паттерн, достаточно самостоятельный чтобы стать отдельным скиллом

---

## ФАЗА 2 — Классификация и фильтрация

Для каждого найденного сигнала определи:

**Scope:**
- `global` — применимо в любом проекте → предложить добавить в `~/.kilo/rules/`
- `project` → в `.kilo/rules/[category].md` или в `AGENTS.md`
- `task` — разовое, не переиспользуемое → **отсеять**

**Тип изменения:**
- `rule` → правило в `.kilo/rules/[category].md` или новая секция в `AGENTS.md`
- `skill` → новый скилл в `.kilo/skills/[name]/SKILL.md`
- `learning` → запись в `.kilo/learnings/LEARNINGS.md`
- `error` → запись в `.kilo/learnings/ERRORS.md`

**Фильтры — отсеивай:**
- Разовая правка без явного паттерна
- Уже присутствует в существующих файлах (проверяй по аудиту из Шага 1)
- Проблема внешнего сервиса вне зоны контроля
- Слишком расплывчато ("будь аккуратнее", "пиши лучше")

**Правило промоции в AGENTS.md** — сигнал становится rule в AGENTS.md если:
- Встречается 3+ раз в сессии, ИЛИ
- Пользователь сказал "всегда", "никогда", "запомни", ИЛИ
- Касается критического поведения (git, файловая система, архитектура, API контракты)

## ФАЗА 4 — Интерактивное применение

Жди ответа пользователя и действуй по таблице:

| Ввод | Действие |
|---|---|
| `y` | Применить все предложения |
| `n` | Ничего не менять, завершить |
| `1,3,5` | Применить только эти номера |
| `skip SKILL` | Применить всё кроме SKILL-предложений |
| `edit 2` | Показать предложение 2, принять правку, затем применить |

### Правила записи для каждого типа:

**RULE → AGENTS.md или .kilo/rules/[category].md**
1. Прочитай весь целевой файл
2. Найди подходящую секцию (или создай новую с заголовком)
3. Вставь правило кратко: 1-2 строки, конкретно и без расплывчатости
4. Покажи diff перед записью, жди подтверждения

**SKILL → .kilo/skills/[name]/SKILL.md**
1. Проверь по аудиту что скилла с таким `name` нет
2. Создай директорию `.kilo/skills/[name]/`
3. Создай `SKILL.md` по шаблону (см. ниже)
4. Имя директории = значение поля `name` в frontmatter (строчные буквы, цифры, дефисы)

**LEARNING → .kilo/learnings/LEARNINGS.md**
1. Если файла нет — создай (см. шаблон ниже)
2. Добавь новую запись в конец файла между маркерами
3. Никогда не удаляй и не редактируй существующие записи

**ERROR → .kilo/learnings/ERRORS.md**
1. Если файла нет — создай
2. Добавь запись в конец, никогда не удаляй существующие

## ФАЗА 5 — Итоговый отчёт

```text
✅ HARNESS EVOLVED
  + [N] правил → AGENTS.md / .kilo/rules/
  + [N] скилл(ов) создан → .kilo/skills/
  + [N] записей → .kilo/learnings/LEARNINGS.md
  + [N] записей → .kilo/learnings/ERRORS.md

⚠️  Важно: начни новую сессию чтобы KiloCode подхватил новые скиллы.
    Новые rules вступят в силу после перезагрузки конфига.
```

## Шаблон нового скилла (.kilo/skills/[name]/SKILL.md)

```markdown
---
name: [skill-name]
description: [ОДНА строка до 1024 символов — что делает, когда применять, триггерные слова]
metadata:
  extracted-from-session: [YYYY-MM-DD]
  pattern-key: [kebab-case]
  recurrence-count: 1
---

# [Название]

## Проблема
[Что происходит, симптомы, когда возникает]

## Когда применять
- [Конкретная ситуация 1]
- [Конкретная ситуация 2]

## Решение

### Шаг 1
[Действие]

### Шаг 2
[Действие]

## Проверка
[Как убедиться что сработало]

## Ограничения
[Где этот подход не работает]
```

**Критические правила KiloCode для SKILL.md:**
- `name`: строчные буквы, цифры, дефисы. Максимум 64 символа. Должен точно совпадать с именем директории
- `description`: одна строка (никакого multiline YAML!), максимум 1024 символа
- `metadata` (опционально): произвольные key-value пары — используй для трекинга recurrence-count
